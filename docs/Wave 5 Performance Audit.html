P-01
Duplicate Google Fonts request — extra render-blocking round-trip
F05.1.1 User Input Monitoring Form.html · lines 8 & 11
High
⛔ Render-blocking ~2–4 KB extra transfer

What's wrong

F05.1.1 loads Google Fonts twice. The first <link> (line 8) requests Poppins 400/500/600/700 + Barlow Semi Condensed + Inter — the standard set for the whole tool. A second <link> (line 11) requests Poppins again with weight 300 added. Weight 300 is not used anywhere in the design system.

Both requests are render-blocking. The browser must establish a connection to fonts.googleapis.com, fetch the CSS descriptor, then fetch the font binaries before painting text. Doing this twice for the same family stalls first paint unnecessarily.

Proposed fix

Delete the second <link> entirely. No other screen loads weight 300 and it is not in the token spec. The preconnect hints on lines 6–7 already cover the single correct request.

<link rel="stylesheet" href="https://fonts.googleapis.com/css2?
  family=Poppins:wght@300;400;500;600;700&display=swap" />
↑ remove entirely — duplicate, weight 300 unused

P-04
CLS risk — scene-logo.png missing width on all 9 navbars
All screens · <nav class="sitenav"> · footer in F00
Medium
📐 CLS — sticky nav layout shift on every page

What's wrong

Every screen renders the nav logo as:

<img src="../assets/scene-logo.png"
     alt="SCeNe Coalition"
     style="height:44px" />

Only height is set; no width attribute is present. Until the browser downloads the image and reads its natural dimensions, it cannot compute the aspect ratio or reserve the correct horizontal space. This causes a layout shift (CLS) in the sticky navbar the moment the image loads — visible as horizontal jitter on slower connections.

The footer logo in F00 has the same issue and also lacks loading="lazy".

Proposed fix

Add an explicit width to every logo <img>. Measure the asset's natural pixel width and set both attributes so the browser can pre-compute the intrinsic ratio:

<img src="../assets/scene-logo.png"
     alt="SCeNe Coalition"
     style="height:44px" />
<img src="../assets/scene-logo.png"
     alt="SCeNe Coalition"
     width="176" height="44"
     style="height:44px;width:auto" />

The actual width value must match the image's true pixel width. Additionally, the footer instance in F00 should gain loading="lazy" since it is always below the fold.

P-05
No <link rel="preload"> for LCP background image on F00 Home
F00 Home.html · home.css .hero-bg / .overview-bg
Medium
⛔ LCP delay — hero image discovered late

What's wrong

The F00 Home hero renders its full-bleed background via CSS:

/* home.css */
.hero-bg {
  background: url(assets/hero-bg.jpg)
    center 30% / cover no-repeat;
}

CSS background-image resources are not discoverable by the browser's preload scanner — they are only fetched after the stylesheet has been parsed and the matching rule applied to a DOM element. On F00, this is the Largest Contentful Paint candidate. The image starts downloading later than it could, extending LCP.

The same applies to assets/overview-bg.jpg used by .overview-bg, which is above the fold on most viewports.

Proposed fix

Add a <link rel="preload"> hint for the hero image in F00 Home.html's <head>, placed before the CSS links so the browser queues the fetch as early as possible:

<link rel="preload"
      href="../assets/hero-bg.jpg"
      as="image" />
<link rel="stylesheet" href="../tokens.css" />
<link rel="stylesheet" href="../home.css" />

Optionally add a second preload for overview-bg.jpg. Only do this on F00 — these hints are page-specific and should not be added to every screen.

P-06
Duplicate CSS reset block in home.css and dashboard.css
home.css · lines 1–7 & dashboard.css · lines 1–10
Low
Minor duplicate CSS weight No rendering impact

What's wrong

Six identical CSS rules are copy-pasted in both files:

  • * { box-sizing: border-box }
  • html { scroll-behavior: smooth }
  • body { -webkit-font-smoothing: antialiased }
  • img { display: block; max-width: 100% }
  • a { color: inherit; text-decoration: none }
  • button { font-family: inherit }

Since every screen already imports tokens.css first, this is the right home for global resets. The duplication adds dead weight and means a future change to any reset requires edits in multiple files.

Proposed fix

Move all six rules to the bottom of tokens.css under a /* ── Global reset ── */ comment, then remove the duplicate blocks from both home.css and dashboard.css. All screens will still inherit the resets via their existing tokens.css import.

Risk: Low. The rules are identical so specificity and cascade order are unchanged. Confirm no screen relies on the reset being declared after a page-specific rule (unlikely given all screens import tokens.css first).

P-07
nav-mobile.js runs three full SVG-harden DOM traversals on every page load
nav-mobile.js · lines 55–60 · all screens
Low
Two unnecessary timer-based traversals on 7 of 9 screens

What's wrong

The hardenSvgs() function does a full querySelectorAll('svg:not([aria-hidden])...') scan across the whole document. boot() calls it three times:

hardenSvgs();              // immediately
setTimeout(hardenSvgs, 400);  // +400ms
setTimeout(hardenSvgs, 1200); // +1200ms

The 400ms and 1200ms passes exist to catch JS-rendered icons on F03 and F05.1.1. However, they fire unconditionally on every screen including the purely static ones (F00, F07, F04, F01 etc.), causing two redundant full-document traversals per page view.

Proposed fix

Gate the delayed passes on the presence of a JS-rendered root element (e.g. #root, which only F03 and F05.1.1 mount):

setTimeout(hardenSvgs, 400);
setTimeout(hardenSvgs, 1200);
if (document.getElementById('root')) {
  setTimeout(hardenSvgs, 400);
  setTimeout(hardenSvgs, 1200);
}

Alternatively, use a MutationObserver on the #root element to harden SVGs only when new nodes are inserted, eliminating timers entirely.

P-08
Auth modal styles inlined in F00 instead of living in auth.css
F00 Home.html · inline <style> block near end of body
Low
~1.1 KB uncacheable inline CSS on the landing page

What's wrong

F00 already links auth.css in its <head>. Despite this, there is a separate <style> block (~50 lines / ~1.1 KB) embedded near the bottom of the <body> that declares the auth modal's overlay and animation rules: .auth-overlay, .auth-modal, .auth-close, body.auth-open, and button.login.

Inline styles cannot be cached independently; they are re-parsed on every visit. They also split style authority for the same component across two locations, making future edits error-prone.

Proposed fix

Move all rules from the inline <style> block into auth.css. Check for rule overlap with what is already in auth.css before merging — there may be partial duplication. Remove the inline <style> block from F00 entirely.

Affects: F00 only. F07 Login uses auth.css directly with no inline override, so no change needed there.

P-09
Repeated inline SVG paths in F01 Step 5 benefit cards (~9× duplication)
F01 Interactive Map.html · Step 5 "Benefit by Pathway" section
Low
~2–3 KB repeated markup in the largest HTML file (54.8 KB)

What's wrong

F01 is the heaviest HTML file at 841 lines / 54.8 KB. Step 5 of the Data Analyser renders three accordion pathway cards (Protect, Manage, Restore), each containing a "Triple benefit contribution" section with three .tbc-row items. The Nature icon path, People icon path and Climate icon path are each written out in full three times (once per pathway card) — nine total inline SVG declarations for three unique icons.

Similarly, the pathway header icons (shield, refresh, leaf) are duplicated between Step 4 and Step 5.

Proposed fix

Use SVG <symbol> + <use> to define each icon once and reference it by ID:

<svg style="display:none" aria-hidden="true">
  <symbol id="ic-nature" viewBox="0 0 24 24"
          stroke-width="1.7" fill="none"
          stroke-linecap="round">
    <path d="M12 21V11M12 13C8 13 5 10 5 6
             c4 0 7 3 7 7Z"/>
  </symbol>
  <!-- … other icons … -->
</svg>

<!-- then each usage becomes: -->
<svg class="tbc-ic" aria-hidden="true">
  <use href="#ic-nature"/>
</svg>

This is worthwhile in F01 specifically given its size; the saving on other screens would be negligible.

P-10
Inline onmouseover/onmouseout handlers on breadcrumb links
F03, F05, F06, F05.1.1 · <nav class="crumbbar">
Low
No measurable perf impact, but CSP-hostile and fragile

What's wrong

The breadcrumb navigation on four screens uses inline event attributes to change link colour on hover:

<a href="…"
   onmouseover="this.style.color=
     'var(--text-primary)'"
   onmouseout="this.style.color=
     'var(--text-base-second)'">

Inline handlers are evaluated as script strings on each event, resist minification, and will be blocked by any Content-Security-Policy: script-src 'self' header added later. A CSS rule is faster, cacheable, and maintainable.

Proposed fix

Remove all inline handlers and replace with a single CSS rule. The .crumbbar class is already present on the element:

/* in nav.css or tokens.css */
.crumbbar a {
  color: var(--text-base-second);
  transition: color var(--motion-fast);
}
.crumbbar a:hover {
  color: var(--text-primary);
}

Also move the breadcrumb's inline style="display:flex;align-items:center;…" layout properties into the same CSS rule to keep HTML clean.

P-11
Footer logo in F00 not lazy-loaded
F00 Home.html · <footer class="footer">
Low
Unnecessary early image fetch on a long-scroll page

What's wrong

F00 is the tool's longest page (576 lines, multiple tall sections). The footer contains a second <img> for scene-logo.png:

<img src="../assets/scene-logo.png"
     alt="SCeNe Coalition"
     style="height:44px;…" />

This footer image is always off-screen on page load (it is at the very bottom of a multi-section page). Without loading="lazy", the browser eagerly fetches it even though the user may never scroll that far, competing for bandwidth with the above-the-fold hero image.

Proposed fix

Add loading="lazy" to the footer logo. Combine with the width/height fix from P-04:

<img src="../assets/scene-logo.png"
     alt="SCeNe Coalition"
     width="176" height="44"
     loading="lazy"
     style="height:44px;width:auto" />

The nav logo (at the top of the page) should not be lazy-loaded — it is always in the initial viewport.
