Q-01
Search filter index desyncs after project deletion — wrong rows hidden
F04 Dashboard.html · search input handler · ~line 296
High
🐛 Functional bug after deletionFixed

What's wrong

The search handler matches DOM children to data by array index:

[...rows.children].forEach((tr, i) => {
  tr.style.display =
    projects[i].name.toLowerCase()
      .includes(q) ? '' : 'none';
});

After a project row is deleted via tr.remove(), the projects array is not spliced. rows.children[0] now corresponds to what was projects[1], so every subsequent search hides the wrong rows.

Proposed fix

Match by a data-name attribute stamped on each row at render time, rather than by index:

// At render time (already present on delete buttons):
<tr data-name="${p.name}" ...>

// Search handler:
[...rows.children].forEach((tr, i) => {
  tr.style.display =
    projects[i].name...includes(q)...
});
[...rows.children].forEach(tr => {
  const name = (tr.dataset.name || '')
    .toLowerCase();
  tr.style.display =
    name.includes(q) ? '' : 'none';
});

Apply the same pattern to grid cards.

Q-03
Profile dropdown IIFE duplicated verbatim across 7 screens — belongs in nav-mobile.js
F01, F03, F04, F04.1, F05, F05.1.1, F06 · inline <script> after <nav>
Medium
7× repeated script blockFixed

What's wrong

Every authenticated screen has this identical inline block immediately after the nav:

(function(){
  const wrap =
    document.getElementById('profileWrap');
  if(!wrap)return;
  document.getElementById('profileBtn')
    .addEventListener('click', e => {
      e.stopPropagation();
      wrap.classList.toggle('open');
    });
  document.addEventListener('click',
    () => wrap.classList.remove('open'));
  document.addEventListener('keydown',
    e => { if(e.key==='Escape')
      wrap.classList.remove('open'); });
})();

nav-mobile.js is already loaded by all 7 screens and runs on DOMContentLoaded — the perfect home for this behaviour.

Proposed fix

Add a initProfileDropdown() function inside nav-mobile.js's boot():

function initProfileDropdown() {
  const wrap =
    document.getElementById('profileWrap');
  if (!wrap) return;
  document.getElementById('profileBtn')
    .addEventListener('click', e => {
      e.stopPropagation();
      wrap.classList.toggle('open');
    });
  document.addEventListener('click',
    () => wrap.classList.remove('open'));
  document.addEventListener('keydown',
    e => { if (e.key === 'Escape')
      wrap.classList.remove('open'); });
}

// inside boot():
initProfileDropdown();

Then delete the inline <script> block from all 7 screens. Risk: nav-mobile.js uses DOMContentLoaded — verify the nav element exists by then (it will, since all 7 screens place nav before body scripts).

Q-04
<tr onclick> / <article onclick> navigation bypasses keyboard and middle-click
F04 Dashboard.html · table rows and grid cards
Medium
Keyboard nav brokenNo middle-click / Ctrl+clickFixed

What's wrong

Both the table view (<tr onclick="window.location.href=...">) and grid view (<article onclick="window.location.href=...">) use onclick for whole-row navigation. <tr> and <article> are not focusable by default — keyboard users pressing Tab/Enter cannot navigate to a project. Middle-click or Ctrl+click to open in a new tab also doesn't work.

Proposed fix

Wrap the project name cell in an <a> tag (the primary call-to-action cell) so native link behaviour is preserved. Keep onclick only on the actions column for stop-propagation:

<tr onclick="window.location.href=
  'F04.1 Project Detail.html'">
  <td>...project name...</td>
<tr>
  <td>
    <a href="F04.1 Project Detail.html"
       class="row-link">
      ...project name...
    </a>
  </td>

Q-05
#metaSaveBar has display:none written twice — gap and align-items unreachable
F04.1 Project Detail.html · line 145
Low
Confusing inline style bugFixed

What's wrong

The save bar's inline style string is:

style="display:none;margin-top:14px;
       display:none;gap:10px;
       align-items:center"

display:none appears twice. The second occurrence shadows the first, and gap + align-items follow the second display:none. When JS sets display='flex' at runtime the flex layout works, but the intent is unclear and the double declaration is a maintenance hazard.

Proposed fix

style="display:none;margin-top:14px;
       display:none;gap:10px;
       align-items:center"
style="display:none;margin-top:14px;
       gap:10px;align-items:center"

Remove the first display:none; (the duplicate). The element remains hidden; JS still overrides it to flex correctly.

Q-08
?step=3 query parameter in F05 links is silently ignored
F04.1 Project Detail.html · 2 links to F05 · lines ~468, 856
Low
Dead query param — always opens Step 1Fixed

What's wrong

Two links from F04.1 navigate to F05 Create Monitoring Plan.html?step=3 implying the user should land on step 3 (Indicators). F05 does not read URLSearchParams — it always initialises at step 1, ignoring the parameter entirely.

Proposed fix (two options)

Option A — implement the intent: add URL param reading to F05's init:

const urlStep = parseInt(
  new URLSearchParams(location.search)
    .get('step'));
if (urlStep >= 1 && urlStep <= 3)
  goStep(urlStep);

Option B — remove the parameter if deep-linking is not yet implemented, to avoid implying behaviour that doesn't exist.
