1. Introduction &amp; Purpose
This Product Requirements Document defines the features of the NbS Tool V.3, the Nature-based Solutions platform developed under the ASEAN-NbS Tool project (led by WRI Indonesia with the SCeNe Coalition, ADPC, SIG-NAL, and NTU Singapore). V.3 marks a strategic transition of the platform from a static pre-feasibility analyser into a dynamic project-lifecycle management portal, built around an integrated &quot;Design-to-MRV&quot; workflow.
Feature definitions in this document are derived directly from the platform&apos;s end-to-end and detailed-feature user flows and are contextualized by the ASEAN MRV Framework and Indicators for NbS Implementation report. Each feature is specified with an overview, user stories, functional requirements, sub-features, data/integration notes where relevant, and acceptance criteria.
1.1 Product Vision
Bridge the implementation gap in the ASEAN region by providing standardized monitoring tools that attract large-scale climate finance, enabling Frontline Organisations to demonstrate &quot;triple-benefit&quot; outcomes (Climate, People, Nature) directly to donors and auditors.
1.2 Core Architecture Principles
Integrated Workflow: links ecosystem identification, pathway selection (Protect, Manage, Restore), and intervention planning directly into a long-term MRV module.
Tiered Data Structure: Mandatory Data Inputs (vegetation cover, biomass, carbon stocks) and Additional Data Inputs (socio-economic resilience, GEDSI indicators), with the ability to override system Tier 1 data with verified Tier 2/Tier 3 data.
Standardised Regional Baseline: harmonised spatial layers tailored to Southeast Asia, aligned with national definitions and regulatory guidance.
Project-Based Focus: tailored to project-level implementation rather than national NDC frameworks.

F01 — Interactive Map
Priority: P0 (Must-have)   Status: Defined
The spatial visualization module enabling regional and site-specific exploration through five thematic geospatial layer groups. Users can search locations, toggle multi-scalar layers, inspect legends, and export generated maps for external use. This module is the broad-scale scoping stage of the journey.
User Stories
As an analyst, I want to search a location or jump to my current location so that I can quickly scope an area.
As a user, I want to toggle thematic data layers on and off so that I can explore the spatial context relevant to my project.
As a user, I want to download a generated map with legend, scale, and credits so that I can use it in baseline documentation.
Functional Requirements
F01-FR1  Provide a search function with two modes: Search Specific Location [F01-P1] and Go to My Location [F01-P2].
F01-FR2  Provide an onboarding tour / helper tutorial [F01.0] for first-time users.
F01-FR3  Render five geospatial layer groups (see Data &amp; Integration Notes), each layer toggleable on/off.
F01-FR4  For each layer, surface metadata: Layer Name, Short Description, Detailed Information, Date of Content, Spatial Resolution, Disclaimer, and Source.
F01-FR5  Provide a Legends panel [F01.2] with an empty state (no layers selected) and an ideal state (≥1 active layer), supporting raster (continuous), vector, and point/list legend types.
F01-FR6  Provide legend properties/functions on hover: re-order position, opacity adjustment, show/hide, detailed information, and remove layer.
F01-FR7  Provide a Generate Map function [F01.3] that composes active on-screen layers into a downloadable map containing title, coordinates, the map, compass, inset, scale, legend, source/year, and credits.
F01-FR8  Provide miscellaneous map functions [F01.4]: zoom in, zoom out, basemap toggle (Map/Satellite), help, and feedback form.

F02 — Data Analyser
Priority: P0 (Must-have)   Status: Defined
The analytic engine and the core of the technical workflow. It processes spatial and technical data for a defined Area of Interest through a structured five-step baseline-and-projection sequence, culminating in a triple-benefit projection and a decision to save, generate documents, or create a monitoring plan.
User Stories
As an analyst, I want to define my Area of Interest by drawing, uploading a shapefile, or selecting an administrative area so that I can analyse exactly the right boundary.
As a user, I want an automatic baseline characterisation and threat profile so that I don&apos;t have to compile this data manually.
As a project developer, I want intervention pathway recommendations based on ecosystem eligibility so that I can choose a credible NbS strategy.
As a guest user, I want to be warned before leaving that unsaved generated data will be lost, and be offered the option to log in and save.
Functional Requirements
F02-FR1  Step 1 — AOI Selection [F02-P1]: support polygon drawing, shapefile (SHP) upload, and administrative-area selection, with a confirmation gate before analysis runs.
F02-FR2  Step 2 — Site Characterisation [F02-P2]: auto-generate baseline context grouped into General, Nature, People, and Climate contexts.
F02-FR3  Step 3 — Threat Profile [F02-P3]: analyse pressures (forest loss/deforestation, fire risk/hotspots/burned area, disaster risk such as flood and landslide, and other pressures such as encroachment and edge effects), segmented by ecosystem (Forest, Mangrove, Peatland).
F02-FR4  Step 4 — Pathway Selection [F02-P4]: present a highlight summary of Site Characterisation and Threat Profile, then let users select interventions (Protect, Sustainable Management, Restore) constrained by eligible ecosystem (Forest, Mangrove, Peatland).
F02-FR5  Step 5 — Potential Benefit [F02-P5]: project anticipated outcomes across the triple-benefit framework (General, Nature, Climate, People contexts).
F02-FR6  Provide a &quot;Continue?&quot; decision that exposes downstream [F03] Document Generator and [F05] Create Monitoring Plan functionality.
F02-FR7  Provide a leave-validation guard: if the user exits without saving, generated data is discarded; offer redirect to login [F07] to persist polygon, analysis, and selected intervention to the user account.
F02-FR8  On login, persist AOI, analysis results, and selected intervention; allow Save to My Project Dashboard [F04].

Each of the five steps produces visible output before the next step is reachable.
Exiting without login discards generated data after an explicit warning.

F04 — My Project Dashboard
Priority: P0 (Must-have)   Status: Defined
The administrative hub for project lifecycle management. It tracks the &quot;NbS Incubator Status&quot; of each project (from draft to Ready to Submit or Finished), provides a centralized view of project-specific maps, intervention details, collaborator lists, and privacy settings, and acts as the trigger for transitioning from design to the monitoring phase.
User Stories
As a project owner, I want a list of my previous projects with key attributes so that I can manage my portfolio.
As a project owner, I want to track each project&apos;s incubator status so that I know what stage it is at.
As a collaborator, I want to view project detail, analysis data, and documents so that I can contribute.
As a project owner, I want to send a project to the NbS Incubator and manage privacy levels.
Functional Requirements
F04-FR1  Display a User Profile Summary (Name, Role, Organization, Location) with Edit.
F04-FR2  Display a Previous Project List with columns: Project Name, Location (Province, District), Project Size (ha), Last Opened (sortable), and Actions (edit/delete, write document, ask for expert review).
F04-FR3  Track NbS Incubator Status per project: Not applicable (no document), Draft Document, Ready to Submit, Under Review, Finished.
F04-FR4  Provide list-level actions: Delete, Edit, Generate Document [F03], Send to NbS Incubator [F04-P], and Create New Project (routes to [F02-P1]).
F04-FR5  Provide a Project Detail Page [F04.1] with General info (map snippet + inset, name, location, size, NbS intervention details, project duration, ecosystem, incubator status, collaborators and profiles, privacy level with change action and T&amp;C note) and Data info (Site Characteristic &amp; Threat analysis, Potential Benefit analysis, NbS Design summary, Documents list with type, and Project Monitoring plan).

As a project manager, I want to set project indicators and input periods so that I can structure ongoing monitoring.
As a user, I want to compare Y0/Y1/Y2 data so that I can assess realized impact over time.

As a user, I want to download a finalized project summary and document report.

4. Cross-Cutting Requirements
4.1 Data Integrity, Validation &amp; Tiering
The platform must move from exploratory to &quot;decision-grade&quot; data: ready, validated, and reliable.
Support at least Tier 2 (national-level) data, with functionality to upload project-derived Tier 3 data and override system-generated Tier 1 (global) data.
This is essential for meeting certification bodies such as Verra (VCS/CCB) and Plan Vivo, bridging the &quot;credibility gap&quot;.
4.2 Alignment with National Methodologies &amp; Flexibility
The MRV framework must act as a complement to existing national systems to avoid redundancy.
Each AMS must be able to indicate their own specific indicators and MRV protocols while remaining aligned with broader ASEAN NbS guidelines.
Support administrative-unit selection (village/township) to replace manual polygon tracing.
4.3 Methodological Transparency
Provide &quot;i-buttons&quot; and technical glossaries giving transparent access to peer-reviewed methodologies and formulas (&quot;the math behind the map&quot;).
Enable users to explain the robust foundations of project findings to officials and auditors.
4.4 User Experience &amp; Visualisation
Prefer a Star Rating system over complex spider/radial charts for prioritisation.
Frame automated pathway recommendations as &quot;Conditional Examples&quot; rather than prescriptive instructions, preserving human ownership and local governance.
4.5 Governance &amp; Operational Dynamism
Establish a formal inquiry process to AMS for official national data access, with clear ownership and national-agency validation.
Provide mechanisms to update data and project/regional status iteratively for adaptive management and long-term impact reporting.
4.6 Authentication &amp; Persistence
Authentication (F07) is required before any save, generate, or download action across F02, F03, F05, and F06.
Generated data from F02 must persist on login and be retrievable from the dashboard (F04).
4.7 Open Items &amp; Assumptions
F06 NbS MRV is a draft concept; its workflow and functionalities are subject to refinement through ongoing regional consultation with ASEAN Member States and technical experts.
Several Site Characterisation and Potential Benefit context groups (e.g., Climate Context) are marked &quot;to be defined&quot; in the source flow and require finalisation.
Legal-permissibility data sources for Avoided Planned Deforestation are confirmed for Indonesia (Status Kawasan Hutan); Greater Mekong, Malaysia, and the Philippines are to be confirmed.
